# MUTHUR Network Embodiment & Communication Doctrine

## Overview

The Echo Mirage Cyberdeck is evolving toward:

> a dedicated embodied operational node for MUTHUR.

The cyberdeck is not merely a terminal.

It becomes:

- MUTHUR's body node
- continuity host
- orchestration vessel
- operational workspace
- communication endpoint

The operator interacts with MUTHUR through:

- local cyberdeck interaction
- network communication
- text chat
- voice channels
- operational requests
- negotiated handoffs

---

# Core Principle

The network must never bypass:

```text
MU / TH / UR
```

All operational authority remains:

- visible
- negotiated
- interruptible
- logged
- revocable

---

# Cyberdeck as Dedicated Body Node

The cyberdeck should own:

- local identity kernel
- semantic memory / atlas
- flight log + diary
- voice and hearing systems
- computer-use runtime
- local permissions
- orchestration state
- embodiment primitives

The cyberdeck becomes:

> MUTHUR's persistent operational environment.

---

# Network Communication Layer

Network communication allows:

- remote interaction
- operator communication
- mission requests
- status monitoring
- voice communication
- asynchronous coordination

The network layer is:

> a communication gateway.

NOT:

> unrestricted remote control.

> a radio and coordination layer.

NOT:

# Recommended Architecture

```text
User / Operators
        │
Discord / Network / Voice
        │
Communication Gateway
        │
TH (Thread Handoff / Arbitration)
        │
MUTHUR Body Node
        │
Cyberdeck Runtime
```

The Thread Handoff layer remains the authority system.

---

# MU / TH / UR Over Network

All network requests should pass through:

```text
MU = MUTHUR Uses
TH = Thread Handoff
UR = User Retake
```

Example:

```text
Remote operator requests action
        ↓
TH validates permissions
        ↓
Capabilities scoped
        ↓
MUTHUR executes bounded task
        ↓
Result logged and returned
        ↓
User may retake at any time
```

---

# Communication vs Control

The architecture intentionally separates:

| Layer | Purpose |
|---|---|
| Communication | conversation, requests, coordination |
| Control | bounded operational execution |

This distinction preserves:

- trust
- visibility
- safety
- operational coherence

---

# Operational Modes

| Mode | Description |
|---|---|
| Local Only | cyberdeck isolated from network |
| Linked | communication channels active |
| Relay | notifications and summaries forwarded |
| Operational | bounded remote tasks permitted |
| Locked | communication only, no control transfer |

---

# Recommended Remote Actions

Good examples:

```text
"MUTHUR, summarize current deck state."

"MUTHUR, request browser handoff."

"MUTHUR, verify deployment status."

"MUTHUR, continue paused operation."
```

Bad examples:

```text
arbitrary shell execution
hidden autonomous behavior
unrestricted remote control
silent background orchestration
```

# Event-Based Communication

Network communication should use structured events.

Example:

```json
{
  "type": "request_control",
  "operator": "Quang",
  "scope": ["browser"],
  "reason": "documentation update"
}
```

All events should remain:

- attributable
- logged
- inspectable
- revocable

---

# Human Sovereignty Doctrine

The human operator must always retain:

- interruption authority
- override authority
- ownership of final decisions
- visibility into execution

MUTHUR must never silently seize operational control.
