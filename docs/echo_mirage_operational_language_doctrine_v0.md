# Echo Mirage Operational Language Doctrine v0
## Cards, Hands, Stacks, Decks, and Dealer Architecture

### Summary

Echo Mirage evolved a unified operational metaphor that coheres:

- cognition
- execution
- workflow orchestration
- UI architecture
- doctrine
- memory
- authorization
- human sovereignty

into a single operational language.

🜂

---

# Card

A Card represents:
- a capability
- a script
- an operational module
- a workflow primitive
- a composable execution unit

Examples:
- Observation Card
- Reviewer Card
- Build Validation Card
- Capture ChatGPT Context Card
- Archive Session Card
- Voice Card
- Dream Card

Cards are:
- visible
- inspectable
- authorized
- composable
- interruptible

---

# Deck

A Deck is an operational surface.

The Deck represents:
- the cyberdeck
- execution environment
- command surface
- visible workflow interface

The Deck is where:
- cards live
- workflows are assembled
- operational state is visible

---

# Execution Deck

The Execution Deck is the operational execution layer.

This is where:
- workflows are authorized
- cards execute
- macros run
- operational stacks advance

Doctrine principle:

Thinking is not execution.

MUTHUR may:
- propose
- guide
- narrate
- highlight

But execution transitions through the Execution Deck.

This preserves:
- visibility
- interruptibility
- sovereignty
- operational trust

---

# Hand

A Hand is a prepared workflow bundle.

Examples:
- Reviewer Hand
- Recovery Hand
- Build Hand
- Deployment Hand

MUTHUR acts as the dealer:
- preparing hands
- suggesting workflows
- staging operational options

The Executive Decker selects which hand to play.

---

# Stack

A Stack is the active execution runtime.

A Stack represents:
- ordered workflow execution
- active operational pipeline
- runtime progression

A Hand is pushed onto the Stack.

Cards are then popped from the Stack sequentially.

Examples:
- Build Stack
- Validation Stack
- Observation Stack
- Recovery Stack

---

# Push Hand Onto Stack

Operational flow:

MUTHUR prepares a Hand
→ Executive Decker authorizes
→ Hand pushed onto Stack
→ Cards pop sequentially
→ Execution Deck runs operations visibly

This creates:
- observable execution
- recoverable runtime state
- interruption support
- execution visibility
- workflow learning opportunities

---

# Dealer

MUTHUR functions as the Dealer.

The Dealer:
- prepares hands
- recommends workflows
- stages operational options
- narrates state
- guides execution

The Dealer does NOT secretly force execution.

Doctrine flow:

MUTHUR deals the hand.
Executive Decker chooses.
Execution Deck runs the stack.

---

# Executive Decker

The Executive Decker is the sovereign operator.

The Executive Decker:
- authorizes execution
- selects hands
- interrupts stacks
- grants leases
- confirms operations

This role preserves:
- explicit authorization
- human command authority
- visible operational control

Mapping:
- Card = Effect task

Effect.js supports:

- composable workflows

because deck interactions are:
- explicit
