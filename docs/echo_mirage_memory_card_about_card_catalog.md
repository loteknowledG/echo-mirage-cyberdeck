# What Is The Card Catalog?

The Card Catalog is the organizational and retrieval layer of Cyberdeck.

Purpose:
- search cards
- retrieve cards
- organize cards
- compose decks
- preserve operational knowledge
- discover workflows
- manage runtime capability

The Card Catalog is not merely a file browser.

It is:
- an operational registry
- a doctrine archive
- a workflow library
- a capability inventory
- a runtime retrieval system

---

# Why The Catalog Matters

As Cyberdeck grows, cards will multiply rapidly.

Without organization:
- workflows become lost
- operators duplicate logic
- doctrine drifts
- runtime becomes incoherent
- orchestration becomes unmanageable

The Card Catalog preserves:
- discoverability
- coherence
- portability
- operational literacy

# Human-Readable First

The Card Catalog must remain understandable to humans.

Operators should be able to:
- inspect cards
- understand relationships
- trace workflows
- search operational intent
- remix workflows
- teach others

The catalog should not become:
- opaque enterprise metadata sludge
- unreadable GUID forests
- hidden automation graphs

---

# Semantic Relationships

Cards should support relational topology.

Examples:

```text
REQUEST_CODEX_REVIEW
  depends_on:
    - CAPTURE_BUILDER_RESULT

PREPARE_RECOVERY_HAND
  related_to:
    - RETRY_BUILDER
```

The Card Catalog should eventually support:
- graph relationships
- semantic retrieval
- nearest-context search
- operational topology mapping
