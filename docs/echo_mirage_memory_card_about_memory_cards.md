# What Is A Memory Card?

A Memory Card is a portable operational knowledge artifact used by Cyberdeck systems.

Memory Cards preserve:
- doctrine
- workflows
- runtime semantics
- implementation coherence
- recovery procedures
- operational philosophy
- tutorials
- continuity context
- lessons learned

A Memory Card is:
- inspectable
- readable
- versionable
- portable
- loadable
- shareable

Memory Cards are intended to be human-readable first.

---

# Core Doctrine

Cyberdeck uses cards as portable operational objects.

Memory Cards are the continuity layer.

They preserve:
- understanding
- intent
- structure
- operational meaning

Memory Cards prevent:
- implementation drift
- terminology drift
- incoherent orchestration
- forgotten runtime semantics

# Runtime Philosophy

Markdown declares.
Runtime interprets.
Scripts execute.
Receipts preserve.

Memory Cards are not raw executable code.

Instead:
- Markdown describes operational meaning
- Runtime systems interpret card semantics
- JavaScript/TypeScript/Python perform execution
- Receipts preserve historical outcomes

# Memory Card Structure

A Memory Card may contain:
- title
- doctrine
- pseudocode
- routes
- metadata
- examples
- implementation notes
- references
- runtime semantics

Example:

```md
# REQUEST_CODEX_REVIEW

clicks: 2
risk: caution

actions:
  - capture(builder_result)
  - request(codex_review)

routes:
  PASS: archive_outcome
  REVISE: prepare_recovery_hand
```

# Click Semantics

Clicks represent operational actions.

A 1-click card:
- minimum guaranteed single operational action

A 2-click card:
- minimum guaranteed two operational actions

Higher click count means:
- more workflow commitment
- more orchestration depth
- more runtime exposure

Higher click count does NOT automatically mean greater power.

---

# Turns

Turns represent runtime occupancy duration.

A low-click card may still occupy runtime for many turns due to:
- retries
- waiting
- quorum review
- operator intervention
- provider latency
- recovery flows
